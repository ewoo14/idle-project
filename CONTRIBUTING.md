# 기여 규칙 (CONTRIBUTING)

본 저장소는 **멀티 에이전트 PR 워크플로우** 로 운영됩니다. 모든 기여자(인간 개발자 포함)는 아래 규칙을 반드시 지켜주세요.

---

## 1. 한글 우선 원칙

- **모든 문서, PR 본문, PR/이슈 코멘트, 리뷰, 커밋 메시지 본문은 한글** 로 작성합니다.
- 영문 허용 범위: 코드 식별자(변수/함수/클래스/파일명), 커밋 제목 prefix(`feat:`, `fix:`, `chore:` 등), 외부 라이브러리 명, URL.
- 한글 표현이 어색해질 정도의 외래어(예: `Pawn`, `Actor`, `WebSocket`)는 그대로 사용하되 첫 등장 시 짧게 부연 설명합니다.

## 2. PR 워크플로우 (필수)

```
[1] PM(Claude) 기획 → PR 생성
[2] Codex CLI 구현팀 (7개 파트) → PR 브랜치에 푸시
[3] Claude 리뷰팀 (7개 파트) → PR 코멘트로 한글 리뷰
[4] TM(Team Manager) → 종합 리뷰 + fix 지시 게시
[5] Codex 구현팀 → 리뷰/fix 반영
[6] Claude 리뷰팀 → 재검토 → TM 종합
[7-A] fix 없음 → PM 보고 → CI 통과 확인 → PM 종합 소견 → 머지
[7-B] fix 있음 → [5] 로 루프
```

자세한 흐름은 [`docs/workflow/01-pm-codex-claude-loop.md`](docs/workflow/01-pm-codex-claude-loop.md) 참고.

## 3. 브랜치 명명

- `plan/<번호>-<주제>` — PM 기획 PR (예: `plan/01-pm-foundation`)
- `feat/<번호>-<주제>` — 기능 구현 (예: `feat/02-combat-core`)
- `fix/<번호>-<주제>` — 버그 수정
- `chore/<주제>` — 잡무 (의존성, 설정, 문서 보강)
- `docs/<주제>` — 문서 전용 변경

## 4. 커밋 메시지

```
<type>: 한글 한 줄 요약 (50자 이내)

한글 본문 (선택). 무엇을 / 왜 / 어떻게 를 간결히.
관련 이슈가 있으면 마지막 줄에 `관련: #123`.
```

`<type>` 목록: `feat | fix | chore | docs | refactor | test | style | perf | ci | build`.

## 5. PR 본문 (필수 항목)

PR 템플릿이 자동 적용됩니다. 다음 섹션을 모두 채워주세요.

1. **요약** — 무엇을 만들/고치는지 한글로.
2. **배경/동기** — 어떤 기획서/이슈에 근거하는지.
3. **변경 내역** — 디렉터리/모듈 단위로.
4. **테스트 / 검증** — 어떻게 동작을 확인했는지.
5. **체크리스트** — 워크플로우 단계 진행 표시.

## 6. 리뷰 규칙

- 리뷰 코멘트는 **구체적** 으로 작성합니다. 추상적인 "좋아 보입니다" 금지.
- 지적은 **근거(기획서/문서/표준)** 와 함께. 가능하면 수정 제안 코드/문구를 첨부합니다.
- 리뷰 우선순위 표시: `[블로커]`, `[중요]`, `[권장]`, `[질문]`, `[칭찬]`.
- 7개 파트 리뷰어가 각자 파트에 대해 코멘트하고, TM 이 PR 본문에 종합 리뷰를 게시합니다.

## 7. CI 통과 필수

- 모든 PR 은 GitHub Actions CI 가 녹색이어야 머지됩니다.
- 추가 검증: 문서 lint, 서버 단위 테스트, 클라이언트 빌드 스모크 (마일스톤에 따라 활성화).

## 8. 권한 / 보안

- Codex CLI 와 Claude 에이전트 모두 **전체 권한** 으로 운영됩니다 (`--dangerously-bypass-approvals-and-sandbox` 등).
- 단, 다음은 금지: 시크릿 커밋, `git push --force origin main`, GitHub 토큰/`.env` 노출.

## 9. 문서화 의무

다음 변경 시 문서를 함께 갱신해야 합니다.
- API 시그니처 변경 → `docs/api/`
- DB 스키마 변경 → 마이그레이션 + `docs/api/schema.md`
- 게임 규칙/밸런스 변경 → `docs/planning/05-balance-philosophy.md` 또는 관련 기획서
- 워크플로우 변경 → `docs/workflow/`

문서가 빠진 PR 은 TM 이 자동으로 `[블로커]` 처리합니다.
