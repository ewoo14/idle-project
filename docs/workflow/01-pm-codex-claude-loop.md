# 멀티 에이전트 PR 루프 — 상세 정의

> 모든 PR 은 본 흐름을 따릅니다. 예외 없음.

---

## 1. 단계 정의

### [1] PM 기획 (Plan)
- 주체: **PM (Claude)**
- 산출물:
  - 기획서 / 작업 명세 (`docs/planning/...` 또는 PR 본문)
  - PR 생성 (`plan/...` 또는 `feat/...` 브랜치)
  - 7개 파트별 작업 지시 (PR 본문 또는 별도 코멘트)
- 종료 조건: PR 이 열렸고, Codex 구현팀이 작업을 시작할 수 있는 상태.

### [2] Codex 1차 구현 (Implement)
- 주체: **Codex CLI 7개 에이전트**
  - 디자이너 / 스토리 / 퀘스트 / 캐릭터·아이템·능력치 / 밸런스 / 백엔드·DB / QA
- 산출물: PR 브랜치에 커밋 (`codex(<role>): ...` 메시지)
- 권한: `--dangerously-bypass-approvals-and-sandbox` (사용자 명시)
- 종료 조건: 모든 해당 파트 작업이 푸시됨. PR 본문 체크리스트 [2] 체크.

### [3] Claude 1차 리뷰 (Review)
- 주체: **Claude 리뷰 7개 에이전트**
  - 각자 자기 파트만 리뷰 (디자이너 → 디자인, 백엔드 → 백엔드 코드/SQL)
- 산출물: PR 코멘트 (라인 코멘트 + 종합 코멘트)
  - 우선순위 태그 필수: `[블로커]`, `[중요]`, `[권장]`, `[질문]`, `[칭찬]`
  - 근거 인용 의무 (기획서 / 문서 / 표준)
- 종료 조건: 7파트 모두 리뷰 코멘트 게시. PR 본문 체크리스트 [3] 체크.

### [4] TM 1차 종합 (Synthesize)
- 주체: **TM (Tech Manager, Claude)**
- 산출물: PR 본문 코멘트 — 종합 리뷰
  - 7파트 리뷰 요약 (블로커/중요만 집중)
  - 충돌하는 리뷰 의견 조정안
  - fix 우선순위 리스트
  - 다음 단계 지시 (Codex 에게 무엇을 고치라고)
- 종료 조건: 종합 코멘트 게시. PR 본문 체크리스트 [4] 체크.

### [5] Codex 2차 fix
- 주체: **Codex 7개 에이전트**
- 입력: 4단계 TM 종합 + 3단계 개별 리뷰 모두
- 산출물: fix 커밋 (`codex(<role>): fix - <리뷰 인용>`)
- 종료 조건: TM 이 지시한 모든 블로커/중요 항목이 반영됨. 체크리스트 [5] 체크.

### [6] Claude 2차 검토 (Re-review)
- 주체: **Claude 리뷰 7개 에이전트**
- 산출물: 코멘트 — "[승인]" 또는 추가 지적
- 종료 조건: 7파트 재검토 완료. 체크리스트 [6] 체크.

### [7] TM 2차 종합 + 분기
- 주체: **TM**
- 산출물: PR 본문 종합 코멘트 — fix 여부 판정
- **분기**:
  - **fix 없음** → PM 에게 보고 ("PM, 이 PR 머지 준비 완료") → 단계 [8] 로
  - **fix 있음** → 단계 [5] 로 되돌아감 (루프)

### [8] PM 종합 소견 + 머지 (Merge)
- 주체: **PM**
- 사전 조건:
  - TM 이 "fix 없음" 보고
  - GitHub Actions CI 모두 통과 (녹색)
- 산출물:
  - PR 본문 코멘트 — **PM 종합 소견** (필수)
    - 이 PR 이 무엇을 달성했는지
    - 리뷰 핵심 쟁점과 해결 결과
    - 다음 PR 예고
  - merge 실행 (`gh pr merge --squash --delete-branch`)
- 종료 조건: main 에 머지, 브랜치 삭제, 다음 PR 진행.

## 3. 운영 규칙

### 3.1 동시성
- Codex 7파트는 **동일 PR 에 병렬 작업**. 충돌은 git rebase / 작은 커밋으로 회피.
- Claude 7파트 리뷰는 **읽기 전용** — 직접 커밋 금지. (단, TM 의 종합 fix 는 별도 fix 커밋 허용)

### 3.2 라벨
- `step/2-codex-impl`, `step/3-claude-review`, `step/4-tm-synthesize`, `step/5-codex-fix`, `step/6-claude-recheck`, `step/7-tm-synthesize-2`, `step/8-pm-merge`
- 라벨은 PM/TM 이 단계 전환 시 갱신.
- 추가 라벨: `type/feat`, `type/fix`, `type/chore`, `priority/p0~p3`, `risk/high|med|low`

### 3.3 시간 제한 (가이드)
- 한 단계 24~48시간 이상 정체 시 PM 이 개입 (재지시 / 파트 단순화).
- 루프 [5]-[6]-[7] 가 3회 이상 돌면 PM 이 스코프 축소 검토.

### 3.4 PR 분할
- 7파트 중 일부 파트만 변경되는 PR 도 허용 (예: 백엔드만, 밸런스만).
- 단, 워크플로우의 단계 [2]~[8] 은 동일하게 적용 (해당 파트 리뷰만 필수).

### 3.5 핫픽스
- 운영 장애 시 PM 단독 권한으로 `hotfix/` 브랜치 → 직접 머지 가능.
- 머지 후 24h 이내 사후 리뷰 PR 필수.

## 4. 도구 호출

### 4.1 Codex CLI 호출 헬퍼
```powershell
# codex/scripts/invoke.ps1 참고
codex exec `
  --dangerously-bypass-approvals-and-sandbox `
  -C "C:\game\idle game\repo" `
  --skip-git-repo-check `
  -p "<역할 프로필>" `
  "<역할별 작업 지시>"
```

### 4.2 Claude 리뷰 호출 (Claude Code 내부)
- PM/TM 이 Agent 도구로 sub-agent 호출.
- 각 리뷰 sub-agent 는 본 워크플로우 + 자기 파트 명세를 시스템 프롬프트로 받음.

## 5. 문서 / 메모리 동기화

- 본 워크플로우 변경 시:
  1. 본 문서 갱신
  2. `CLAUDE.md` (저장소 루트 자동 로드) 갱신
  3. PM/TM 메모리 (`project_workflow.md`) 갱신
  4. 모든 에이전트 시스템 프롬프트의 워크플로우 첨부 부분 갱신
  5. 변경 PR 본문에 영향 범위 명시

---

## 6. 위반 시

- 단계 누락 PR 은 TM 이 `[블로커]` 처리.
- PM 임의 머지는 즉시 리버트 + 재머지 PR 필수.
- 한글 외 리뷰는 TM 이 리뷰 무효 처리 (재작성 요구).
